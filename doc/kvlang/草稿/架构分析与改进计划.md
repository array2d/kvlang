# kvlang 架构分析与改进计划

> 分析时间：2026-07-10  
> 分析范围：全部 Go 源码（5110 行）  
> 结论：**核心层优雅，执行引擎层混乱且低效**

---

## 一、整体层次结构

```
┌─────────────────────────────────────────────────────┐
│  cmd/kvlang          CLI 入口 (load / serve / vet)  │
├─────────────────────────────────────────────────────┤
│  internal/parser     .kv 文本 → AST                 │
│  internal/lower      AST → 指令序列 (控制流降低)    │
├─────────────────────────────────────────────────────┤
│  internal/layoutcode 加载 + 运行时混合层  ← 问题核心 │
├─────────────────────────────────────────────────────┤
│  internal/kvcpu      执行引擎（execute + control）  │
│  internal/vtype      VType 命名空间分发接口          │
│  internal/op/dispatch 后端路由（op-plat / heap-plat）│
│  internal/builtin    原生内建算子                    │
├─────────────────────────────────────────────────────┤
│  internal/vthread    VThread 状态管理               │
│  internal/sched      调度器（Pick / Run）            │
│  internal/keytree    KV 路径常量中心                 │
│  internal/kvspace    KVSpace 接口（Redis 实现）      │
└─────────────────────────────────────────────────────┘
```

## 二、做得好的地方

| 模块 | 评分 | 说明 |
|------|------|------|
| `keytree` | ⭐⭐⭐⭐⭐ | 所有 KV 路径集中一处，修改路径无需搜索全库 |
| `kvspace` 接口 | ⭐⭐⭐⭐⭐ | 干净的 6 个方法，Redis/Mock 可互换，测试友好 |
| `vtype` 命名空间 | ⭐⭐⭐⭐ | `tensor.*` / `str.*` 前缀分发，消灭了 `isFunctionCall` 的 KV 查询 |
| `lower` pass | ⭐⭐⭐⭐ | `if/for/while/break/continue` → `call/br/return` 降低干净，逻辑与执行引擎解耦 |
| `builtin.Op` 接口 | ⭐⭐⭐⭐ | 内建算子可插拔，每个算子自注册 |
| `keytree` 路径规范 | ⭐⭐⭐⭐ | `/src/<pkg>/<name>`、`/func/<pkg>/<name>/[i,j]`、`/vthread/<vtid>` 设计清晰 |
| execute.go 分发 | ⭐⭐⭐⭐ | 四路静态 switch（control / native / vtype / user-func），无 KV 分类查询 |

## 三、架构问题

### 3.1 死代码（立即可删）

| 文件 | 死代码 | 说明 |
|------|--------|------|
| `internal/cache/cache.go` | `KVCache`、`NewKVCache` | 整包从未被任何生产代码调用 |
| `internal/op/instruction.go` | `DecodeFromCache` | 函数定义存在，无调用方 |
| `kvcpu/controlflow.go` | `handleBr` | 只有 `brToCall` 被使用，`handleBr` 从未调用 |
| `kvcpu/controlflow.go` | `funcRoot`、`blockPC` | 定义存在，从未被外部调用 |

### 3.2 `layoutcode.go` — 职责混乱的 God Object

同一个包（239 行）混合了两个完全不同的阶段：

```
Load 时（编译期）:           Run 时（运行期）:
  WriteFunc                    HandleCall
  WriteBody                    HandleReturn
  RegisterBlocks               copyFunc
  parseSig（重复）             parseSig（重复）
```

`HandleCall` 和 `HandleReturn` 是**运行时控制流**，应属于 `kvcpu` 层；  
`WriteFunc` 是**编译/加载**，应属于 `compiler` 层。  
两者混合导致：修改执行模型必须同时理解加载细节，修改加载格式必须同时理解运行时状态机。

**理想拆分：**

```
internal/compiler/      WriteFunc, WriteBody, RegisterBlocks
internal/kvcpu/         HandleCall, HandleReturn, copyFunc（或直接执行）
```

### 3.3 AST 侵入了存储层

```go
// ast/ast.go — AST 节点直接写 KV，破坏了分层
func (i *Instruction) SetKV(kv kvspace.KVSpace, prefix string, idx *int) { ... }
func (fn *Func) Register(kv kvspace.KVSpace, pkg string) error { ... }
```

`Stmt` 接口因此携带了 `SetKV(kv kvspace.KVSpace, prefix string, idx *int)` 方法，  
导致 `IfStmt`、`ForStmt`、`WhileStmt`、`BreakStmt`、`ContinueStmt` 都有空实现 `{}`。  
AST 是纯数据结构，不应知道存储层。`SetKV` 应由外部 `compiler` 包负责。

### 3.4 `layoutcode.parseSig` 重复了 `parser.ParseSignature`

两段逻辑几乎相同，分布在相距 3 个包的文件里。应直接调用 `parser.ParseSignature`。

### 3.5 `vtype` 作为跨层胶水包

```
vtype/tensor.go → dispatch（知道后端细节）
vtype/str.go    → builtin（知道内建细节）
```

`vtype` 本应是纯接口定义包，现在却了解两个子系统。  
改进：将具体 Exec 逻辑移入 `kvcpu`，`vtype` 只负责接口注册和 `Lookup`。

## 四、正确性缺陷

### 4.1 `sched.Pick` — TOCTOU 竞态

```go
// 读和写之间不原子，两个 worker 可能同时看到 status=init
val, err := kv.Get(fullKey)   // 读 status=init
...
kv.Set(fullKey, data)         // 写 status=running
```

两个 worker 同时通过第一行，都写 `running`，同一任务执行两次。  
**修复**：使用 Redis `SET NX` 或 Lua 原子 CAS。

### 4.2 `vthread.Set` 丢失 `Mode` 字段

```go
func Set(ctx, kv, vtid, pc, status string) {
    s := VThread{PC: pc, Status: status}  // Mode、Error 字段丢失
    ...
}
```

VThread 创建时设置 `Mode: "single"`，但每次 `Set` 都重建对象，导致整个执行过程中 `Mode` 始终是零值。  
**修复**：`Set` 先 `Get` 再合并，或改为接收完整 `VThread` 对象。

### 4.3 `incrVtid` 非原子

```go
func incrVtid(kv kvspace.KVSpace) string {
    val, _ := kv.Get(keytree.SysVtidCounter)  // Get
    n, _ := strconv.Atoi(val)
    n++
    kv.Set(keytree.SysVtidCounter, strconv.Itoa(n))  // Set
}
```

多 VM 并发时会生成重复 vtid。  
**修复**：使用 Redis `INCR` 命令（单条原子操作）。

### 4.4 `inBody` 检测脆弱

```go
// parser/file.go — 用文本前缀匹配判断是否在函数体内
if strings.HasPrefix(line, bl.FirstLine()) { inBody = true }
```

函数签名行的文本内容作为"是否进入函数体"的判断依据，  
当函数签名包含特殊字符或出现同名前缀时会误判。  
**修复**：应基于缩进层级或显式的 `{}`/`end` 标记。

## 五、性能问题

### 5.1 `copyFunc` — 每次函数调用 O(N×slots) 次 KV 写入

每次调用一个函数，都将 `/func/<pkg>/<name>/[i,j]` 全量复制到 `/vthread/<vtid>/[n,0]/[i,j]`：

```
调用 poly3（9条指令，每条约3个槽位）的代价：
  List(/func/arith/poly3)              = 1  次 LIST
  每条指令读 opcode + reads + writes   = 9×4 = 36 次 GET
  写入 /vthread/<vtid>/...             = 9×4 = 36 次 SET
  ─────────────────────────────────────────────────
  合计 ≈ 73 次 KV 操作，仅为"复制函数体"
```

根本原因：执行模型是"把指令拷贝进 vthread 栈再执行"，而非"在 `/func/` 原地执行，栈帧只存绑定"。

**理想执行模型：**

```
CallFrame {
    FuncPath string            // 指向 /func/<pkg>/<name>
    PC       int               // 当前指令偏移
    Bindings map[string]string // 参数名 → vthread 路径
}
```

执行时直接从 `/func/` 读取指令，用 `Bindings` 做参数替换，无需 copy。

### 5.2 每条指令最小 KV 开销

```
vthread.Get     → 1 GET   （读 PC+status）
op.Decode       → 1 MGET  （最多 21 个 key）
执行工作        → N GET/SET
vthread.Set     → 1 SET   （写 PC+status）
──────────────────────────────────────────
最少 3 个 round-trip，每条指令
```

对于 `A * B -> ./C` 这样的简单标量乘法：需要 **5 次 KV 操作**才能完成。

### 5.3 `mainWatcher` 轮询

```go
ticker := time.NewTicker(1 * time.Second)  // 每秒轮询
val, err := kv.Get(keytree.FuncMain)
```

`/func/main` 是"函数就绪信号 key"，应使用 `Watch`（底层 BLPOP）事件驱动，  
而非每秒轮询。

| 模块 | 评分 | 主要问题 |

## 七、改进优先级

### 🔴 P0 — 立即修复（改动小，消除 bug）

1. **删 `cache` 包**：`internal/cache/cache.go` 整包删除
2. **删死代码**：`DecodeFromCache`、`handleBr`、`funcRoot`、`blockPC`
3. **修 `vthread.Set` 丢 Mode**：Set 前先 Get 合并，或接收完整结构体
4. **修 `incrVtid` 原子性**：改用 Redis `INCR`

### 🟠 P1 — 重要修复（中等改动，消除竞态和低效）

5. **`sched.Pick` 原子化**：SET NX 或 Lua CAS
6. **`mainWatcher` 改事件驱动**：Watch / BLPOP 替代 ticker
7. **删 `layoutcode.parseSig`**：直接调用 `parser.ParseSignature`

### 🟡 P2 — 架构重构（大改动，提升优雅性）

8. **拆分 `layoutcode`**：  
   - `internal/compiler/` ← WriteFunc, WriteBody, RegisterBlocks  
   - `HandleCall / HandleReturn` 移入 `kvcpu/`

9. **AST 去 KV 依赖**：  
   - 删除 `ast.Stmt.SetKV` 接口方法  
   - 删除 `ast.Func.Register`  
   - 由 `compiler` 包负责序列化 AST → KV

10. **执行模型：CallFrame 代替 copyFunc**：  
    - 栈帧只存 `{FuncPath, PC, Bindings}`  
    - 直接从 `/func/` 读取指令执行  
    - 消除每次调用的 O(N) KV copy
