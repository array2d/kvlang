#### 3.1.1 `lexer` 包的额外问题：switch 语句逻辑错误

整包已是死代码，但其 keyword switch 还有 bug，记录于此以防将来被重新启用：

```go
// lexer/lexer.go — 缩进与 Go 语法不符，导致错误行为
case op.OpIf:           // 空 body（Go 无 fallthrough）→ "if" 不产生任何 token
    case "else":        // 实际是 switch 顶层的 case
        tokens = append(tokens, Token{Kind: Else, Value: word})
    tokens = append(tokens, Token{Kind: If, Value: word})
    // 当 word=="else"：同时 append Else + If → 错误

case op.OpFor:          // 空 body → "for" 不产生任何 token
    case op.OpWhile:
    case op.OpBreak:
    case op.OpContinue:
        tokens = append(tokens, Token{Kind: Ident, Value: word})
    tokens = append(tokens, Token{Kind: For, Value: word})
    // 当 word=="continue"：append Ident + For → 错误
```

### 3.2 正确性缺陷

#### 3.2.1 `mainWatcher` 创建 vthread 绕过 struct，Mode 字段丢失

```go
// serve.go:142 — mainWatcher（serve 模式）：Mode 字段缺失
kv.Set(base, `{"pc":"[0,0]","status":"init"}`)

// serve.go:24 — executeEntry（单次模式）：正确
st := vthread.VThread{PC: "[0,0]", Status: "init", Mode: "single"}
data, _ := json.Marshal(st)
kv.Set(keytree.VThread("run"), data)
```

P0 修复了 `vthread.Set` 保留 Mode，但 serve 模式的 vthread 从创建起 Mode 就是空串。
**修复**：mainWatcher 改用 `vthread.VThread` struct 序列化，或调 `vthread.CreateVThread`。

#### 3.2.2 `vthread.Set` 已修复（P0）

原问题：`Set(vtid, pc, status)` 重建 `VThread{}` 会丢弃 Mode 字段。
现状：已改为 Get + merge，Mode 字段正确保留。

### 3.3 逻辑冗余

#### 3.3.1 `strVType.Exec` 中的 `IsNativeOp` 检查永远为 false

```go
// vtype/str.go
func (s strVType) Exec(...) error {
    if builtin.IsNativeOp(inst.Opcode) { // ← 永远 false，死分支
        return builtin.Native(...)
    }
    // error
}
```

原因：`execute.go` dispatch 顺序是 `IsNativeOp`（case 2）在 `vtype.Lookup`（case 3）之前。
能到达 `strVType.Exec` 的 opcode 必然已通过 `!IsNativeOp` 过滤，所以 `IsNativeOp` 检查在此永远为 false。

#### 3.3.2 `parseParamList` vs `parseParamListRaw` 近乎重复

两个函数逻辑完全相同（约 50 行各一份），唯一差别是是否调 `stripQuotes`。
应合并为一个带 `stripQuotes bool` 参数的版本。

### 3.5 架构层面问题（P2，大改动）

#### 3.5.1 `layoutcode` 混合了编译期和运行时

同一个包（202 行）混合了两个完全不同的执行阶段：

```
编译期（load 时调用）：        运行时（execute 时调用）：
  WriteFunc                       HandleCall
  WriteBody                       HandleReturn
  RegisterBlocks                  copyFunc
```

修改执行模型时必须同时理解加载细节；修改加载格式时必须同时理解运行时状态机。
**理想拆分**：`internal/compiler/` ← 编译期；`HandleCall/HandleReturn` 移入 `kvcpu/`。

#### 3.5.2 `ast.Stmt.SetKV` 耦合存储层

`Stmt` 接口含 `SetKV(kv kvspace.KVSpace, prefix string, idx *int)`，导致：
- `IfStmt`、`ForStmt`、`WhileStmt`、`BreakStmt`、`ContinueStmt` 全部有空实现 `{}`
- AST 数据结构不得不 import `kvspace` 和 `keytree`

AST 是纯数据结构，不应知道存储层。`SetKV` 应由外部 compiler 包负责。

#### 3.5.3 `copyFunc` O(N×M) KV 操作——每次函数调用的主要开销

```
调用一个 10 指令的函数：
  kv.List(srcPrefix)            = 1  次 SMEMBERS
  每条指令：Get opcode + 10 reads + 10 writes = 21 次 GET
  写入 vthread：同上             = 最多 21 次 SET
  ──────────────────────────────────────────────────
  总计 ≈ 1 + 10×21 + 10×21 = 421 次 KV 操作，只为"复制函数体"
```

根本原因：执行模型是"把指令 copy 进 vthread 栈再执行"，而非"栈帧只存绑定，直接在 `/func/` 执行"。

**理想模型**：
```go
type CallFrame struct {
    FuncPath string            // 指向 /func/<pkg>/<name>
    PC       int               // 当前指令偏移
    Bindings map[string]string // 形参名 → vthread 路径
}
// 执行时直接读 /func/ 指令，用 Bindings 替换参数，无需 copy
```

#### 3.5.4 `/func/main` 一个 key 当两种用途

```
写入触发：{"entry":"pre_main","reads":[],"writes":[]}
mainWatcher 认领后删除，然后写入：{"vtid":"1","status":"executing"}
```

同一个 key 先是"启动信号"，后变成"执行状态"，语义混乱。应拆为两个 key：
- `/func/main`：只作启动触发，认领后永久删除
- `/vthread/<vtid>`：vthread 状态（已有）
