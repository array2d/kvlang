# keytree 最高标准设计

> 父节点：根目录 `最高标准设计.md` → 本文档

---

## 零、一句话定位

**keytree 是 kvlang 的路径系统**：将所有运行时概念映射到 kvspace 的键路径，
就像 Unix 将进程、设备、配置全部映射到文件系统路径一样。

keytree 包只做一件事：**把名字组合成路径**。没有 I/O，没有逻辑，只有字符串。

---

## 一、设计哲学（Unix 类比）

| Unix 文件系统 | kvlang keytree | 原则 |
|-------------|---------------|------|
| `/proc/<pid>/` | `/vthread/<vtid>/` | **所有权编码在路径中**：进程/虚线程的一切属性都在其路径下 |
| `/sys/class/net/eth0/` | `/sys/vm/<id>/` | **层次即结构**：路径结构反映事物的逻辑从属关系 |
| `/dev/tty0` | `/dev/tty/<name>/` | **设备独立命名空间** |
| `/lib/` | `/func/` | **可执行代码**：ELF 二进制 ↔ kvlang IR 文本，格式不同，语义相同 |
| `/proc/sys/kernel/pid_max` | `/vthread/seq` | **系统元数据是普通子项**，无特殊前缀 |

### 四条铁律

1. **`/` 是唯一分隔符**。路径中禁止出现 `:`。
2. **路径即拥有者**。X 的所有属性必须在 X 的路径前缀下，不得散落到其他根。
3. **一个节点可以同时是值和命名空间**。`/vthread/<vtid>` 存状态 JSON，同时有子项 `[0,0]`、`done`。
4. **键是纯绝对路径，各段只含 `[a-z0-9-]`**。路径段中不含 `.`；`.` 仅出现在槽的**值**中（如 `./slot` 相对引用），由执行引擎解释，与 keytree 无关。

## 二、路径全景树（filesystem view）

```
kvspace /
│
├── func/                              ← 可执行代码（类 /lib，IR 文本，与 /src/ 同格式不同阶段，经过编译优化）
│   ├── main                           ← 程序运行请求（JSON，VM 轮询消费）
│   ├── idx/
│   │   └── <funcname>                 ← 反向索引：函数名 → 包路径
│   └── <pkg>/
│       └── <name>                     ← 函数签名字符串（FuncSig.String()）
│           └── [i,j]                  ← 指令槽（i=行号，j=列：0=opcode，+n=write，-n=read）
│
├── src/                               ← 源码原文（ast写入，未经过编译优化）
│   └── <pkg>/
│       └── <name>                     ← 函数完整源码文本
│           └── [i,j]                  ← 指令槽（i=行号，j=列：0=opcode，+n=write，-n=read）
│
├── vthread/                           ← 虚线程运行时（类 /proc，高度动态）
│   ├── seq                            ← vtid 序列计数器（单 key，原子递增）
│   ├── ready                          ← 新 vthread 就绪通知队列（Notify/Watch）
│   └── <vtid>/                        ← vthread 根路径（DelR 整体清理）
│       ├── .pc                        ← 当前 PC（绝对路径字符串，必须以 /vthread/ 开头）
│       ├── .status                    ← init | running | wait | done | error
│       ├── .mode                      ← single | batch
│       ├── .err                       ← 错误码（存在即表示 error 状态）
│       │   └── msg                    ← human-readable 错误描述
│       └── .done                      ← 完成信号（值为 "ok" 或 "error"）
│           └── result                 ← done=ok 时的返回值
│       ├── [i,j]                      ← 当前帧指令槽
│       │   └── [i,j]/                 ← 子调用帧（嵌套 call 展开，理论上无限深度）
│       └── term                       ← 绑定终端名（可选，无则缺失）
│
├── dev/                               ← 设备（类 /dev）
│   └── tty/
│       └── <name>/
│           └── <stream>               ← stream ∈ {stdin, stdout, stderr, pty}
│
└── sys/                               ← 系统基础设施（类 /sys）
    ├── vm/
    │   └── <id>                       ← VM 实例状态 JSON：{status, pid, started_at}
    │       ├── hb                     ← 心跳 JSON：{ts, status, pid}
    │       └── cmd                    ← VM 命令队列（Watch/Notify）
    ├── op/
    │   └── <backend>/                 ← backend 名如 cuda、pytorch、jax
    │       ├── <n>                    ← 第 n 个实例状态（n = "0","1",…）
    │       │   └── cmd                ← 实例命令队列
    │       └── func/
    │           └── <name>             ← 算子函数定义

heap对象的路径，是自由的，交由kvlang的程序员自行配置路径，自行完成heap对象回收
```

## 三、指令槽约定 `[i,j]`

`[i,j]` 是矩阵坐标：`i` = 行（指令序号），`j` = 列（槽角色）。
函数体是一张二维表，行是指令，列是操作数：

```
 i \ j │ … │  -2  │  -1  │  0(op)  │  +1       │  +2  │ … │
───────┼───┼──────┼──────┼─────────┼───────────┼──────┼───┤
   0   │   │   2  │   1  │   add   │ ./result  │      │   │
   1   │   │      │   …  │  return │   …       │      │   │
```

| j | 语义 |
|---|------|
| `0` | opcode |
| `+1`, `+2`, … | 写槽（输出） |
| `-1`, `-2`, … | 读槽（输入） |

槽的**值**（参数名、立即数等）由执行引擎解释，keytree 不感知值的格式。

**子调用帧**：`call foo` 在 PC=`[3,0]` 处展开，形成子路径
`/vthread/<vtid>/[3,0]/`，foo 的指令序列写入该子路径下的 `[i,j]`。

## 四、路径规范（按根分组）

### 4.1 /func/

```
/func/main                  程序运行请求入口
/func/idx/<funcname>        反向索引：函数名 → 包路径
/func/<pkg>/<name>          函数签名字符串
/func/<pkg>/<name>/[i,j]    指令槽
```

### 4.2 /src/

```
/src/<pkg>/<name>           函数完整源码文本
/src/<pkg>/<name>/[i,j]     指令槽
```

### 4.3 /vthread/

```
/vthread/seq                vtid 序列计数器
/vthread/ready              新 vthread 就绪通知队列

/vthread/<vtid>/.pc             当前 PC（绝对路径，必须以 /vthread/ 开头）
/vthread/<vtid>/.status         init | running | wait | done | error
/vthread/<vtid>/.mode           single | batch
/vthread/<vtid>/.err            错误码（存在即表示 error 状态）
/vthread/<vtid>/.err/msg        human-readable 错误描述
/vthread/<vtid>/.done           完成信号，值为 ok | error（Watch 等待）
/vthread/<vtid>/.done/result    done=ok 时的返回值

/vthread/<vtid>/[i,j]           当前帧指令槽
/vthread/<vtid>/[i,j]/[k,l]     子调用帧指令槽（可任意嵌套）
/vthread/<vtid>/term            绑定终端名（可选）
```

> `.` 开头的路径段是引擎保留键（类比 Linux 隐藏文件），kvlang 标识符不能以 `.` 开头，用户代码无法写入。

### 4.4 /dev/

```
/dev/tty/<name>/<stream>    stream ∈ {stdin, stdout, stderr, pty}
```

### 4.5 /sys/

```
/sys/vm/<id>/hb             VM 心跳
/sys/vm/<id>/cmd            VM 命令队列
/sys/vm/<id>/err            VM 错误通知队列（unknown opcode 等）

/sys/op/<backend>/<n>       op 后端第 n 个实例状态
/sys/op/<backend>/<n>/cmd   实例命令队列
/sys/op/<backend>/func/<name>  算子函数定义
```

## 五、路径命名规范

| 规范 | 示例 | 禁止 |
|------|------|------|
| 用户路径段只含 `[a-z0-9-]` | `op-cuda`、`pre-main` | 大写、`_`、`:` |
| 路径分隔：只用 `/` | `/sys/vm/0` | `/sys/vm:0` |
| 路径不以 `/` 结尾 | `/vthread/42` | `/vthread/42/` |
| 数字后缀用裸数字 | `/sys/vm/0`、`/sys/op/cuda/1` | `/sys/vm/vm-0` |
| 保留段名 | `idx`、`seq`、`ready`、`term`、`hb`、`cmd`、`func` | 用户包/函数名避免与保留段名冲突 |
| 引擎保留键 | `.` 开头（如 `.pc`、`.done`） | 类比 Linux 隐藏文件；kvlang 标识符不能以 `.` 开头 |

---

## 六、禁止项

| 编号 | 禁止 | 理由 |
|------|------|------|
| F1 | keytree 包 import 任何 I/O 或业务包 | 它只组合字符串，永远不执行 |
| F2 | 路径中出现 `:` | Unix 路径无 `:`；与 IPv6 地址冲突 |
| F3 | 路径段中间出现 `.`（如 `foo.bar`） | `.` 在键中只允许出现在段的**开头**（引擎保留键），中间位置禁止；`./slot` 相对引用只出现在槽的**值**中 |
| F4 | 路径以 `/` 结尾 | 路径是标识符，不是目录字面量 |
| F5 | 在 keytree 之外硬编码根前缀字符串 | 所有路径只允许经由 keytree 函数生成 |
| F6 | 将同一概念的属性散布在多个根下 | 违反"所有权编码在路径中"原则 |
| F7 | 新增路径函数时不同步更新路径全景树 | 本文档是路径系统的唯一权威 |

---

## 七、为什么不用 Redis keyspace 约定（`:`分隔符）

Redis 社区常用 `user:1001:name` 的冒号约定，但 kvspace 是树形路径系统：

- `KVSpace.List(prefix)` 依赖 `/` 分层语义（类似 `readdir`）
- `Link(target, linkpath)` 是路径级软ln
- `DelR(prefix)` 是路径前缀的递归删除

用 `:` 会破坏 `List` 和 `Link` 的语义一致性。

## 八、完整路径示例（pre-main → calc → add，两次调用）

### 8.1 源代码（.kv）

```kv
def add(a:int, b:int) -> (c:int) {
    c <- a + b
}

def calc(x:int, y:int) -> (z:int) {
    add(x, y) -> z
}

calc(3, 4) -> result
```

### 8.2 加载后：/func/ 静态快照

`kvlang load` 将源码编译并写入 kvspace，路径永久不变：

```
/func/main                      = {"entry":"pre-main","reads":[],"writes":[]}


/func/pre-main             = "def pre-main() -> (result)"
/func/pre-main/[0,-2]      = 4
/func/pre-main/[0,-1]      = 3
/func/pre-main/[0,0]       = calc
/func/pre-main/[0,1]       = ./result
/func/pre-main/[1,0]       = return

/func/calc                 = "def calc(x:int, y:int) -> (z:int)"
/func/calc/[0,-2]          = ./y
/func/calc/[0,-1]          = ./x
/func/calc/[0,0]           = add
/func/calc/[0,1]           = ./z
/func/calc/[1,-1]          = ./z
/func/calc/[1,0]           = return

/func/add                  = "def add(a:int, b:int) -> (c:int)"
/func/add/[0,-2]           = ./b
/func/add/[0,-1]           = ./a
/func/add/[0,0]            = +
/func/add/[0,1]            = ./c
/func/add/[1,-1]           = ./c
/func/add/[1,0]            = return
```

### 8.3 执行期：/vthread/ 动态变化

```
# ── T1：启动 vthread，ln pre-main 帧 ──────────────────────────────
/vthread/seq                    = 1
/vthread/1/                    = {"pc":"[0,0]","status":"running"}
/vthread/1/pre-main             = Link:/func/pre-main

# ── T2：执行 pre-main[0,0]=calc，ln calc 帧 ───────────────────────
/vthread/1/pre-main/calc        = Link:/func/calc
/vthread/1/pre-main/calc/x      = 3
/vthread/1/pre-main/calc/y      = 4

# ── T3：执行 calc[0,0]=add，ln add 帧（栈峰值）───────────────────
/vthread/1/pre-main/calc/add    = Link:/func/add
/vthread/1/pre-main/calc/add/a  = 3
/vthread/1/pre-main/calc/add/b  = 4
# 此刻 /vthread/1 下全部 key（路径深度 = 调用栈深度）：
#   /vthread/1                          {"pc":"[0,0]","status":"running"}
#   /vthread/1/pre-main                 Link:/func/pre-main
#   /vthread/1/pre-main/calc            Link:/func/calc
#   /vthread/1/pre-main/calc/x          3
#   /vthread/1/pre-main/calc/y          4
#   /vthread/1/pre-main/calc/add        Link:/func/add
#   /vthread/1/pre-main/calc/add/a      3
#   /vthread/1/pre-main/calc/add/b      4

# ── T4：执行 add[0,0]=+，写结果 ─────────────────────────────────────
/vthread/1/pre-main/calc/add/c  = 7

# ── T5：add return，DelR 销毁 add 帧 ────────────────────────────────
/vthread/1/pre-main/calc/z      = 7
DEL /vthread/1/pre-main/calc/add
DEL /vthread/1/pre-main/calc/add/a
DEL /vthread/1/pre-main/calc/add/b
DEL /vthread/1/pre-main/calc/add/c

# ── T6：calc return，DelR 销毁 calc 帧 ──────────────────────────────
/vthread/1/pre-main/result      = 7
DEL /vthread/1/pre-main/calc
DEL /vthread/1/pre-main/calc/x
DEL /vthread/1/pre-main/calc/y
DEL /vthread/1/pre-main/calc/z

# ── T7：pre-main return，vthread 完成 ───────────────────────────────
/vthread/1/done                 = {"status":"ok","result":"7"}
/vthread/1                      = {"pc":"[0,0]","status":"done"}
DEL /vthread/1/pre-main
DEL /vthread/1/pre-main/result
```
