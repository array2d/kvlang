# keytree 最高标准设计

> 父节点：根目录 `最高标准设计.md` → 本文档

---

## 零、一句话定位

**keytree 是 kvlang 的路径系统**：将所有运行时概念映射到 kvspace 的键路径，
就像 Unix 将进程、设备、配置全部映射到文件系统路径一样。

keytree 包只做一件事：**把名字组合成路径**。没有 I/O，没有逻辑，只有字符串。

---

## 一、设计哲学（Unix 类比）

| Unix 文件系统 | kvlang keytree | 原则 |
|-------------|---------------|------|
| `/proc/<pid>/` | `/vthread/<vtid>/` | **所有权编码在路径中**：进程/虚线程的一切属性都在其路径下 |
| `/sys/class/net/eth0/` | `/sys/vm/<id>/` | **层次即结构**：路径结构反映事物的逻辑从属关系 |
| `/dev/tty0` | `/dev/tty/<name>/` | **设备独立命名空间** |
| `/lib/` | `/func/` | **可执行代码**：ELF 二进制 ↔ kvlang IR 文本，格式不同，语义相同 |
| `/proc/sys/kernel/pid_max` | `/vthread/seq` | **系统元数据是普通子项**，无特殊前缀 |

### 四条铁律

1. **`/` 是唯一分隔符**。路径中禁止出现 `:`。
2. **路径即拥有者**。X 的所有属性必须在 X 的路径前缀下，不得散落到其他根。
3. **一个节点可以同时是值和命名空间**。`/vthread/<vtid>` 存状态 JSON，同时有子项 `[0,0]`、`done`。
4. **键是纯绝对路径，各段只含 `[a-z0-9-]`**。路径段中不含 `.`；`.` 仅出现在槽的**值**中（如 `./slot` 相对引用），由执行引擎解释，与 keytree 无关。

## 二、路径全景树（filesystem view）

```
kvspace /
│
├── func/                              ← 可执行代码（类 /lib，IR 文本，与 /src/ 同格式不同阶段）
│   ├── main                           ← 程序运行请求（JSON，VM 轮询消费）
│   ├── idx/
│   │   └── <funcname>                 ← 反向索引：函数名 → 包路径
│   └── <pkg>/
│       └── <name>                     ← 函数签名字符串（FuncSig.String()）
│           └── [i,j]                  ← 指令槽（i=行号，j=列：0=opcode，+n=write，-n=read）
│
├── src/                               ← 源码原文（只写入，不修改）
│   └── <pkg>/
│       └── <name>                     ← 函数完整源码文本
│
├── vthread/                           ← 虚线程运行时（类 /proc，高度动态）
│   ├── seq                            ← vtid 序列计数器（单 key，原子递增）
│   ├── ready                          ← 新 vthread 就绪通知队列（Notify/Watch）
│   └── <vtid>                         ← vthread 状态 JSON：{pc, status, mode}
│       ├── [i,j]                      ← 当前帧指令槽
│       │   └── [i,j]/                 ← 子调用帧（嵌套 call 展开，理论上无限深度）
│       ├── done                       ← 完成通知队列（Watch by caller）
│       └── term                       ← 绑定终端名（可选，无则缺失）
│
├── dev/                               ← 设备（类 /dev）
│   └── tty/
│       └── <name>/
│           └── <stream>               ← stream ∈ {stdin, stdout, stderr, pty}
│
└── sys/                               ← 系统基础设施（类 /sys）
    ├── vm/
    │   └── <id>                       ← VM 实例状态 JSON：{status, pid, started_at}
    │       ├── hb                     ← 心跳 JSON：{ts, status, pid}
    │       └── cmd                    ← VM 命令队列（Watch/Notify）
    ├── op/
    │   └── <backend>/                 ← backend 名如 cuda、pytorch、jax
    │       ├── <n>                    ← 第 n 个实例状态（n = "0","1",…）
    │       │   └── cmd                ← 实例命令队列
    │       └── func/
    │           └── <name>             ← 算子函数定义
    └── heap/
        └── <backend>/
            └── <n>                    ← 第 n 个实例状态
                └── cmd                ← 实例命令队列
```

## 三、指令槽约定 `[i,j]`

`[i,j]` 是矩阵坐标：`i` = 行（指令序号），`j` = 列（槽角色）。
函数体是一张二维表，行是指令，列是操作数：

```
 i \ j │ … │  -2  │  -1  │  0(op)  │  +1       │  +2  │ … │
───────┼───┼──────┼──────┼─────────┼───────────┼──────┼───┤
   0   │   │   2  │   1  │   add   │ ./result  │      │   │
   1   │   │      │   …  │  return │   …       │      │   │
```

| j | 语义 |
|---|------|
| `0` | opcode |
| `+1`, `+2`, … | 写槽（输出） |
| `-1`, `-2`, … | 读槽（输入） |

槽的**值**（参数名、立即数等）由执行引擎解释，keytree 不感知值的格式。

**子调用帧**：`call foo` 在 PC=`[3,0]` 处展开，形成子路径
`/vthread/<vtid>/[3,0]/`，foo 的指令序列写入该子路径下的 `[i,j]`。

## 四、路径 API（keytree 包的公开合约）

keytree 包只暴露**纯函数**，没有状态，没有 I/O。每个函数对应路径树中的一个节点。

### 4.1 func/

```go
const FuncRoot  = "/func"
const FuncMain  = "/func/main"       // 程序运行请求

func FuncIdx(name string) string     // /func/idx/<name>
func Func(pkg, name string) string   // /func/<pkg>/<name>
```

### 4.2 src/

```go
const SrcRoot = "/src"

func Src(pkg, name string) string    // /src/<pkg>/<name>
```

### 4.3 vthread/

```go
const VthreadRoot  = "/vthread"
const VthreadSeq   = "/vthread/seq"    // vtid 计数器
const VthreadReady = "/vthread/ready"  // 新 vthread 就绪通知队列

func VThread(vtid string) string                      // /vthread/<vtid>
func VThreadDone(vtid string) string                  // /vthread/<vtid>/done
func VThreadTerm(vtid string) string                  // /vthread/<vtid>/term
func VThreadSlot(vtid, frame string, i, j int) string // /vthread/<vtid>/[i,j] 或子帧
func VThreadFrame(vtid, frame string) string          // /vthread/<vtid>/<frame>/ 子帧前缀
```

### 4.4 dev/

```go
func DevTTY(name, stream string) string  // /dev/tty/<name>/<stream>
```

### 4.5 sys/

```go
const SysRoot = "/sys"

func SysVM(id string) string                    // /sys/vm/<id>
func SysVMHB(id string) string                  // /sys/vm/<id>/hb
func SysVMCmd(id string) string                 // /sys/vm/<id>/cmd
func SysOp(backend, n string) string            // /sys/op/<backend>/<n>
func SysOpCmd(backend, n string) string         // /sys/op/<backend>/<n>/cmd
func SysOpFunc(backend, name string) string     // /sys/op/<backend>/func/<name>
func SysHeap(backend, n string) string          // /sys/heap/<backend>/<n>
func SysHeapCmd(backend, n string) string       // /sys/heap/<backend>/<n>/cmd
```

## 五、路径命名规范

| 规范 | 示例 | 禁止 |
|------|------|------|
| 路径段只含 `[a-z0-9-]` | `op-cuda`、`pre-main` | 大写、`_`、`.`、`:` |
| 路径分隔：只用 `/` | `/sys/vm/0` | `/sys/vm:0` |
| 路径不以 `/` 结尾 | `/vthread/42` | `/vthread/42/` |
| 数字后缀用裸数字 | `/sys/vm/0`、`/sys/op/cuda/1` | `/sys/vm/vm-0` |
| 保留段名 | `idx`、`seq`、`ready`、`done`、`term`、`hb`、`cmd`、`func` | 用户包/函数名避免与保留段名冲突 |

---

## 六、禁止项

| 编号 | 禁止 | 理由 |
|------|------|------|
| F1 | keytree 包 import 任何 I/O 或业务包 | 它只组合字符串，永远不执行 |
| F2 | 路径中出现 `:` | Unix 路径无 `:`；与 IPv6 地址冲突 |
| F3 | 路径段中出现 `.` | `.` 只能出现在槽的值中（`./slot`），不属于键的组成部分 |
| F4 | 路径以 `/` 结尾 | 路径是标识符，不是目录字面量 |
| F5 | 在 keytree 之外硬编码根前缀字符串 | 所有路径只允许经由 keytree 函数生成 |
| F6 | 将同一概念的属性散布在多个根下 | 违反"所有权编码在路径中"原则 |
| F7 | 新增路径函数时不同步更新路径全景树 | 本文档是路径系统的唯一权威 |

---

## 七、为什么不用 Redis keyspace 约定（`:`分隔符）

Redis 社区常用 `user:1001:name` 的冒号约定，但 kvspace 是树形路径系统：

- `KVSpace.List(prefix)` 依赖 `/` 分层语义（类似 `readdir`）
- `Link(target, linkpath)` 是路径级软链接
- `DelR(prefix)` 是路径前缀的递归删除

用 `:` 会破坏 `List` 和 `Link` 的语义一致性。

## 八、完整路径示例（pre-main → calc → add，两次调用）

> `=` 左侧是 kvspace **键**（keytree 负责），右侧是存储的**值**（keytree 不感知其格式）。
>
> 程序：`pre-main` 调用 `calc(3,4)`，`calc` 调用 `add(x,y)`，`add` 计算 `x+y`。

---

### 加载后（/func/ 静止）

```
/func/main          = {"entry":"pre-main","reads":[],"writes":[]}
/func/idx/pre-main  = main
/func/idx/calc      = math
/func/idx/add       = math
```

`/func/main/pre-main`（`def pre-main() -> (result)`）：

```
 i \ j │  -2  │  -1  │   0    │  +1       │
───────┼──────┼──────┼────────┼───────────┤
   0   │   4  │   3  │  calc  │ ./result  │
   1   │      │      │ return │           │
```

`/func/math/calc`（`def calc(x:int, y:int) -> (z:int)`）：

```
 i \ j │  -2   │  -1   │   0    │  +1   │
───────┼───────┼───────┼────────┼───────┤
   0   │  ./y  │  ./x  │  add   │ ./z   │
   1   │       │  ./z  │ return │       │
```

`/func/math/add`（`def add(a:int, b:int) -> (c:int)`）：

```
 i \ j │  -2   │  -1   │   0    │  +1   │
───────┼───────┼───────┼────────┼───────┤
   0   │  ./b  │  ./a  │   +    │ ./c   │
   1   │       │  ./c  │ return │       │
```

---

### T1：vthread 启动，链接 pre-main 帧

```
/vthread/seq           = 1
/vthread/1             = {"pc":"[0,0]","status":"running"}
```

`/vthread/1/pre-main`（Link → `/func/main/pre-main`）：代码透过 Link 可见，无局部槽。

---

### T2：执行 pre-main[0,0]=calc，链接 calc 帧

```
/vthread/1/pre-main/calc  → Link → /func/math/calc
/vthread/1/pre-main/calc/x = 3
/vthread/1/pre-main/calc/y = 4
```

`/vthread/1/pre-main/calc`（代码 via Link，局部参数槽）：

```
 i \ j │  -2   │  -1   │   0    │  +1   │  局部槽        │
───────┼───────┼───────┼────────┼───────┼────────────────┤
   0   │  ./y  │  ./x  │  add   │ ./z   │  x=3, y=4      │
   1   │       │  ./z  │ return │       │                │
```

---

### T3：执行 calc[0,0]=add，链接 add 帧

```
/vthread/1/pre-main/calc/add   → Link → /func/math/add
/vthread/1/pre-main/calc/add/a = 3
/vthread/1/pre-main/calc/add/b = 4
```

**调用栈峰值**——路径深度即栈深度：

```
/vthread/1/pre-main/            → Link → /func/main/pre-main
/vthread/1/pre-main/calc/       → Link → /func/math/calc     x=3, y=4
/vthread/1/pre-main/calc/add/   → Link → /func/math/add      a=3, b=4
```

---

### T4：执行 add[0,0]=+，写结果

```
/vthread/1/pre-main/calc/add/c = 7
```

---

### T5：add return，销毁 add 帧

```
# 读取返回值，写回 calc 的 z 槽
/vthread/1/pre-main/calc/z = 7
# 销毁 add 帧（一次 DelR）
DelR /vthread/1/pre-main/calc/add
```

---

### T6：calc return，销毁 calc 帧

```
# 读取返回值，写回 pre-main 的 result 槽
/vthread/1/pre-main/result = 7
# 销毁 calc 帧
DelR /vthread/1/pre-main/calc
```

---

### T7：pre-main return，vthread 完成

```
DelR /vthread/1/pre-main
/vthread/1/done = {"status":"ok","result":"7"}
/vthread/1      = {"pc":"[0,0]","status":"done"}
```
