# keytree 最高标准设计

> 父节点：根目录 `最高标准设计.md` → 本文档

---

## 零、一句话定位

**keytree 是 kvlang 的路径系统**：将所有运行时概念映射到 kvspace 的键路径，
就像 Unix 将进程、设备、配置全部映射到文件系统路径一样。

keytree 包只做一件事：**把名字组合成路径**。没有 I/O，没有逻辑，只有字符串。

---

## 一、设计哲学（Unix 类比）

| Unix 文件系统 | kvlang keytree | 原则 |
|-------------|---------------|------|
| `/proc/<pid>/` | `/vthread/<vtid>/` | **所有权编码在路径中**：进程/虚线程的一切属性都在其路径下 |
| `/sys/class/net/eth0/` | `/sys/vm/<id>/` | **层次即结构**：路径结构反映事物的逻辑从属关系 |
| `/dev/tty0` | `/dev/tty/<name>/` | **设备独立命名空间** |
| `/lib/` | `/func/` | **可执行代码**：ELF 二进制 ↔ kvlang IR 文本，格式不同，语义相同 |
| `/proc/sys/kernel/pid_max` | `/proc/.seq` | **元数据用点前缀**：`.`开头的子项是系统管理的，不是用户数据 |

### 三条铁律

1. **`/` 是唯一分隔符**。路径中禁止出现 `:`（不得有 `vm:0`、`cmd:op-cuda:0` 这类形式）。
2. **路径即拥有者**。X 的所有属性必须在 X 的路径前缀下，不得散落到其他根。
3. **一个节点可以同时是值和命名空间**。`/proc/<vtid>` 存状态 JSON，同时有子项 `[0,0]`、`.done`。

## 二、路径全景树（filesystem view）

```
kvspace /
│
├── func/                              ← 可执行代码（类 /lib，IR 文本格式，与 /src/ 同格式不同阶段）
│   ├── .main                          ← 程序入口描述符 JSON
│   ├── .idx/
│   │   └── <funcname>                 ← 反向索引：函数名 → 包路径
│   └── <pkg>/
│       └── <name>                     ← 函数签名字符串（FuncSig.String()）
│           └── [i,j]                  ← 指令槽（i=行号，j=列：0=opcode，+n=write，-n=read）
│
├── src/                               ← 源码原文（只写入，不修改）
│   └── <pkg>/
│       └── <name>                     ← 函数完整源码文本
│
├── proc/                              ← 虚线程运行时（类 /proc，高度动态）
│   ├── .seq                           ← vtid 序列计数器（单 key，原子递增）
│   ├── .ready                         ← 新 vthread 就绪通知队列（Notify/Watch）
│   └── <vtid>                         ← vthread 状态 JSON：{pc, status, mode}
│       ├── [i,j]                      ← 当前帧指令槽
│       │   └── [i,j]/                 ← 子调用帧（嵌套 call 展开，理论上无限深度）
│       ├── .done                      ← 完成通知队列（Watch by caller）
│       └── .term                      ← 绑定终端名（可选，无则缺失）
│
├── dev/                               ← 设备（类 /dev）
│   └── tty/
│       └── <name>/
│           └── <stream>               ← stream ∈ {stdin, stdout, stderr, pty}
│
└── sys/                               ← 系统基础设施（类 /sys）
    ├── vm/
    │   └── <id>                       ← VM 实例状态 JSON：{status, pid, started_at}
    │       ├── .hb                    ← 心跳 JSON：{ts, status, pid}
    │       └── .cmd                   ← VM 命令队列（Watch/Notify）
    ├── op/
    │   └── <backend>/                 ← backend 名如 cuda、pytorch、jax
    │       ├── <n>                    ← 第 n 个实例状态（n = "0","1",…）
    │       │   └── .cmd               ← 实例命令队列
    │       └── func/
    │           └── <name>             ← 算子函数定义
    └── heap/
        └── <backend>/
            └── <n>                    ← 第 n 个实例状态
                └── .cmd               ← 实例命令队列
```

## 三、指令槽约定 `[i,j]`

这是 kvlang 的矩阵地址方案，`i` = 指令行号，`j` = 列角色：

| j | 含义 |
|---|------|
| `[i,0]` | opcode（操作名） |
| `[i,1]`, `[i,2]`, … | 写槽（write slot），第 1、2、…个输出 |
| `[i,-1]`, `[i,-2]`, … | 读槽（read slot），第 1、2、…个输入 |

槽里存放的**值**（参数名、立即数等）由执行引擎解释，keytree 不感知值的格式。

**子调用帧**：`call foo` 在 PC=`[3,0]` 处展开，形成子路径
`/proc/<vtid>/[3,0]/`，foo 的指令序列写入该子路径下的 `[i,j]`。

## 四、路径 API（keytree 包的公开合约）

keytree 包只暴露**纯函数**，没有状态，没有 I/O。每个函数对应路径树中的一个节点。

### 4.1 func/（编译代码）

```go
// /func/.main
const FuncMain = "/func/.main"

// /func/.idx/<name>
func FuncIdx(name string) string

// /func/<pkg>/<name>
func Func(pkg, name string) string

// FuncRoot — List 用
const FuncRoot = "/func"
```

### 4.2 src/（源码）

```go
// /src/<pkg>/<name>
func Src(pkg, name string) string

const SrcRoot = "/src"
```

### 4.3 proc/（虚线程）

```go
const ProcRoot   = "/proc"
const ProcSeq    = "/proc/.seq"    // vtid 计数器
const ProcReady  = "/proc/.ready"  // 新 vthread 就绪通知队列

// /proc/<vtid>
func Proc(vtid string) string

// /proc/<vtid>/.done
func ProcDone(vtid string) string

// /proc/<vtid>/.term
func ProcTerm(vtid string) string

// /proc/<vtid>/[i,j]（或子帧路径）
// frame = "" 表示顶层；frame = "[3,0]" 表示 call 子帧
func ProcSlot(vtid, frame string, i, j int) string

// /proc/<vtid>/<frame>/（子帧前缀，用于 List 和批量写入）
func ProcFrame(vtid, frame string) string
```

### 4.4 dev/（设备）

```go
// /dev/tty/<name>/<stream>
func DevTTY(name, stream string) string
```

### 4.5 sys/（系统基础设施）

```go
const SysRoot = "/sys"

// /sys/vm/<id>
func SysVM(id string) string

// /sys/vm/<id>/.hb
func SysVMHB(id string) string

// /sys/vm/<id>/.cmd
func SysVMCmd(id string) string

// /sys/op/<backend>/<n>
func SysOp(backend, n string) string

// /sys/op/<backend>/<n>/.cmd
func SysOpCmd(backend, n string) string

// /sys/op/<backend>/func/<name>
func SysOpFunc(backend, name string) string

// /sys/heap/<backend>/<n>
func SysHeap(backend, n string) string

// /sys/heap/<backend>/<n>/.cmd
func SysHeapCmd(backend, n string) string
```

## 五、路径命名规范

| 规范 | 示例 | 禁止 |
|------|------|------|
| 小写字母 + 数字 + `-`（段内连字符） | `op-cuda`（但不在路径中用 `-` 做分隔） | `Op_Cuda`、`opCuda` |
| 路径分隔：只用 `/` | `/sys/vm/0` | `/sys/vm:0`、`vm_0` |
| 元数据子项：`.` 前缀 | `.hb`、`.cmd`、`.done`、`.seq`、`.ready` | `hb`（易与用户键冲突）|
| 路径不以 `/` 结尾 | `/proc/42` | `/proc/42/` |
| 索引/计数：点前缀在父节点下 | `/proc/.seq` | `/sys/vtid_counter` |
| 段名不含 `:` | `/sys/op/cuda/0` | `/sys/op/cuda:0` |
| 数字后缀用裸数字 | `/sys/vm/0`、`/sys/op/cuda/1` | `/sys/vm/vm-0` |

---

## 六、禁止项

| 编号 | 禁止 | 理由 |
|------|------|------|
| F1 | keytree 包 import 任何 I/O 或业务包 | 它只组合字符串，永远不执行 |
| F2 | 路径中出现 `:` | Unix 路径无 `:`；与 IPv6 地址冲突 |
| F3 | 路径以 `/` 结尾 | Unix 惯例：路径是标识符，不是目录字面量 |
| F4 | 在 keytree 之外硬编码任何 `/func/`、`/proc/` 等根前缀 | 所有路径只允许经由 keytree 函数生成 |
| F5 | 将同一概念的属性散布在多个根下 | 违反"所有权编码在路径中"原则 |
| F6 | 新增路径函数时不同步更新本文档的路径全景树 | 文档是路径系统的唯一权威 |

## 七、为什么不用 Redis keyspace 约定（`:`分隔符）

Redis 社区常用 `user:1001:name` 的冒号约定，但 kvspace 是树形路径系统：

- `KVSpace.List(prefix)` 依赖 `/` 分层语义（类似 `readdir`）
- `Link(target, linkpath)` 是路径级软链接
- `DelR(prefix)` 是路径前缀的递归删除

以上操作全部建立在 `/` 分隔的层次结构上。用 `:` 会破坏 `List` 和 `Link` 的语义一致性。

## 八、完整路径示例（执行一个 `add` 程序）

> 注：`=` 左侧是 kvspace **键**（keytree 负责），右侧是存储的**值**（keytree 不感知其格式）。

```
加载阶段（kvlang load）：
/func/.main                     = {"entry":"pre_main","reads":[],"writes":[]}
/func/.idx/pre_main             = main
/func/.idx/add                  = math
/func/main/pre_main             = "def pre_main() -> ()"
/func/main/pre_main/[0,0]       = add
/func/main/pre_main/[0,-1]      = 1
/func/main/pre_main/[0,-2]      = 2
/func/main/pre_main/[0,1]       = ./result
/func/math/add                  = "def add(a:int, b:int) -> (c:int)"
/func/math/add/[0,0]            = +
/func/math/add/[0,-1]           = ./a
/func/math/add/[0,-2]           = ./b
/func/math/add/[0,1]            = ./c

运行阶段（kvlang serve）：
/proc/.seq                      = 1
/proc/1                         = {"pc":"[0,0]","status":"running","mode":"single"}
/proc/1/[0,0]                   = add
/proc/1/[0,-1]                  = 1
/proc/1/[0,-2]                  = 2
/proc/1/[0,1]                   = ./result

执行完成：
/proc/1                         = {"pc":"[0,0]","status":"done","mode":"single"}
/proc/1/result                  = 3
/proc/1/.done                   = {"status":"ok","result":"3"}
```
