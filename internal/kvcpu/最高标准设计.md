# kvcpu 最高标准设计

> 父节点：根目录 `最高标准设计.md` → 本文档

---

## 零、定位与设计目标

**kvcpu 是以 kvspace 为统一内存的虚拟 CPU**，实现完整的 Fetch-Decode-Execute 循环。

### 设计目标

| 目标 | 含义 |
|------|------|
| **正确性** | 相同输入必须产生相同输出；执行是确定性的 |
| **持久性** | 任意时刻崩溃后重启 VM，vthread 可从最后一个已持久化的 PC 继续 |
| **可观测性** | 所有执行状态（PC、局部变量、调用栈）全部在 kvspace 路径中实时可读 |
| **可扩展性** | 新算子类型通过 vtype 注册，不修改 kvcpu 核心 |
| **并发性** | 多 worker 无 GIL，通过 kvspace 原子语义协调 |
| **热替换** | `/func/` 下的代码随时可被替换，正在执行的帧不受影响 |

## 一、与工业界 VM 的类比

| 概念 | 物理 CPU | CPython | JVM | BEAM (Erlang) | **kvcpu** |
|------|---------|---------|-----|--------------|-----------|
| 指令存储 | 内存线性字节 | `PyCodeObject` 字节数组 | `.class` 字节数组 | BEAM 字节码 | `/func/<pkg>/<name>/[i,j]` |
| PC | 整数寄存器（IP） | `f_lasti` 整数偏移 | PC 寄存器整数 | 指令指针 | **绝对路径字符串**（`/vthread/<vtid>/…`） |
| 帧 | 内存栈段（SP/FP） | `PyFrameObject` 链表 | Stack Frame 链表 | Process heap | kvspace **子树**（路径深度 = 栈深度） |
| 局部变量 | 寄存器/栈偏移 | `fastlocals[]` | 局部变量表 | 寄存器 | 帧内**具名槽** `/frame/varname` |
| 操作数 | 寄存器文件 | 求值栈 PUSH/POP | 操作数栈 | 寄存器 | 指令 `reads`/`writes`（**具名，无匿名栈**） |
| 调用栈 | SP+内存 | `f_back` 指针链 | JVM 栈 | Process 调用链 | 路径层级自然编码 |
| 帧销毁 | RET 弹栈 | `RETURN_VALUE` | `return` | GC | `kv.DelR(framePath)` 一次原子清理 |
| 并发 | 多核硬件 | GIL 串行 | JVM 线程+JIT | Actor 独立堆 | 多 worker 乐观并发，kvspace 仲裁 |
| 崩溃恢复 | 全失 | 全失 | 全失 | 全失 | **重启继续**（PC 已持久化） |
| 热代码替换 | 不支持 | 不支持 | 受限（HotSwap） | 支持（code_change） | **原生支持**（代码即 kvspace 数据） |

**kvcpu 与其他 VM 的本质区别**：帧是 kvspace 子树而非内存对象，路径前缀边界即帧边界，路径深度即调用栈深度。无需额外的栈指针、帧指针、帧链表。

## 二、核心数据结构

### 2.1 vthread —— 执行单元

> 路径规范见 `keytree/最高标准设计.md §4.3`，此处只列执行语义。

vthread 状态全部平铺为独立 key，value 均为标量字符串。
引擎内部 key 位于 `~/` 保留段，用户帧变量始终在 `[i,j]/` 下，两者结构隔离。

status 状态机：

```
init ──pick──► running ──async op──► wait ──resume──► running
                  │                                      │
                  └──────────── done / error ────────────┘
```

### 2.2 帧 (Frame) —— 对应 JVM Stack Frame

帧是代码的执行上下文，由三部分组成：

```
/vthread/<vtid>/<frame>           = Link:/func/<pkg>/<name>   ← ① 代码链接（零拷贝）
/vthread/<vtid>/<frame>/<param>   = <value>                   ← ② 实参 / 局部变量
/vthread/<vtid>/<frame>/done      = <returnVal>               ← ③ 返回值暂存（return 时写入）
```

- **① 代码链接**：`kvspace.Link` 软链到 `/func/`，函数代码零拷贝挂载
- **② 局部变量**：形参在 call 时写入，函数体内可读写，return 时随帧一起 `DelR` 清理
- **③ 返回值**：return 指令将值写入调用方指定的写槽，然后 `DelR` 当前帧

**帧销毁是原子操作**：`kvspace.DelR(/vthread/<vtid>/<frame>)` 一次删除帧的所有键，
等价于 JVM 弹出 Stack Frame，无需逐项枚举。

### 2.3 PC —— 对应 IP 寄存器

**PC 必须是绝对路径，以 `/vthread/` 开头。**

```
/vthread/1/[0,0]                 ← 顶层帧，第 0 条指令
/vthread/1/[0,0]/[1,0]           ← 调用栈深度 2，子帧第 1 条指令
/vthread/1/[0,0]/[1,0]/[2,0]     ← 调用栈深度 3
```

物理 CPU 的 IP 是绝对内存地址，单独持有即可寻址。kvcpu 的 PC 同理——
一个路径字符串完整编码了"在哪个 vthread 的哪一帧执行到哪一行"，
无需额外的 vtid 参数辅助寻址。

```go
NextPC(pc)      // 当前帧下一条：末尾 [i,0] 的 i+1
ParentPC(pc)    // 父帧下一条：去掉末尾路径段后 NextPC
VtidFromPC(pc)  // 提取 vtid：路径第二段
FrameFromPC(pc) // 提取帧路径：去掉末尾 [i,0]
```

### 2.4 指令槽 —— 对应字节码操作数

```
/func/<pkg>/<name>/[i, 0]   = opcode        （必须）
/func/<pkg>/<name>/[i,-1]   = read_operand1 （输入，按需）
/func/<pkg>/<name>/[i,-2]   = read_operand2
/func/<pkg>/<name>/[i,+1]   = write_operand1（输出，按需）
/func/<pkg>/<name>/[i,+2]   = write_operand2
```

kvcpu **无操作数栈**（区别于 JVM/CPython）。每条指令的输入输出完全具名，
等价于 RISC 寄存器模型，但寄存器名是路径字符串。

## 三、Fetch-Decode-Execute 循环

```
Execute(ctx, pc):                      // pc 是绝对路径
  vtid = VtidFromPC(pc)

  loop:
  ┌─── ① FETCH ────────────────────────────────────────────────┐
  │  s  = vthread.Get(vtid)             // 读 {pc, status}     │
  │  if s.status ∈ {done, error} → return                      │
  │  pc = s.pc                          // 绝对路径            │
  └─────────────────────────────────────────────────────────────┘
  ┌─── ② DECODE ───────────────────────────────────────────────┐
  │  inst = op.Decode(kv, pc)                                   │
  │    └─ kv.Gets(pc, pc[i,-1], pc[i,-2], pc[i,+1], …)         │
  │       一次 MGET 批量预取，缺失槽自然截断                     │
  │  if inst.Opcode == "" → vthread.Set(done) ; return          │
  └─────────────────────────────────────────────────────────────┘
  ┌─── ③ EXECUTE ──────────────────────────────────────────────┐
  │  dispatch(inst.Opcode):                                     │
  │    IsControlOp  → handleControl(kv, pc, inst)              │
  │    IsNativeOp   → builtin.Native(kv, pc, inst)             │
  │    vtype.Lookup → VType.Exec(kv, pc, inst)                 │
  │    default      → rewrite as call → handleControl          │
  └─────────────────────────────────────────────────────────────┘
  ┌─── ④ PC ADVANCE ───────────────────────────────────────────┐
  │  每个执行器负责更新 PC：                                    │
  │    顺序指令：vthread.Set(vtid, NextPC(pc), "running")      │
  │    控制跳转：vthread.Set(vtid, targetPC,  "running")       │
  │    异步等待：vthread.Set(vtid, pc,        "wait")          │
  └─────────────────────────────────────────────────────────────┘
```

## 四、Dispatch 优先级（全静态，零 KV 查询）

```
opcode
  │
  ├─① IsControlOp(opcode)            → handleControl
  │    call / return / br / goto
  │
  ├─② builtin.IsNativeOp(opcode)     → builtin.Native
  │    + - * / % == != < > && || ! print sqrt abs int float bool …
  │
  ├─③ vtype.Lookup(opcode) ≠ nil     → VType.Exec
  │    tensor.* → tensorVType
  │    str.*    → strVType
  │    （init() 注册，运行时零查询扩展）
  │
  ├─④ FuncIdx(opcode) ≠ ""           → 用户定义函数
  │    inst = {Opcode:"call", Reads:[opcode, ...inst.Reads], Writes:inst.Writes}
  │    → handleControl
  │
  └─⑤ unknown                        → 错误暂停 + 通知 error worker
       vthread.SetError(vtid, pc, {code:"unknown_opcode", message:opcode})
       kv.Notify(keytree.SysVMErr(vmID), {vtid, pc, opcode})
       worker 退出本次 Execute，vthread 暂停在当前 PC
```

**铁律**：dispatch 不访问 kvspace，全部路由 O(1) 常数时间完成。
新 vtype 通过 `vtype.Register()` 注册，dispatch 自动覆盖，不修改 kvcpu。

> **备注**：`if / for / while / break / continue` 均是高级语法糖，不到达 kvcpu。
> 编译器（lower 层）负责在代码写入 `/func/` 之前将其展开为
> `call / return / br / goto` 的组合。kvcpu 只执行已降级的四条原语，
> 等价于 JVM 字节码中没有 `if`/`for`/`while` 指令——编译器将其编译为 `goto`/`ifeq`。

## 五、控制流完整语义

### 5.1 call / return

```
call(funcName, arg1, arg2) -> ret

调用：
  1. FuncIdx(funcName) → pkg
  2. kv.Link(/func/<pkg>/<funcName>, framePath)    // 挂载代码
  3. kv.Set(framePath/param1, arg1)                // 绑实参
  4. kv.Set(framePath/param2, arg2)
  5. PC = FrameFromPC(pc) + "/[0,0]"              // 跳入

返回：
  1. kv.Set(调用方写槽, returnVal)                 // 回传返回值
  2. kv.DelR(framePath)                            // 销毁帧（一次原子清理）
  3. PC = ParentPC(pc)                             // 跳回父帧
```

### 5.2 br（条件跳转）

```
br(cond, true_label, false_label)

  condVal = ResolveReadValue(kv, pc, cond)
  假值：""、"0"、"false"；其余为真
  选中 label → resolveLabel → call(label) → 进入 label 块帧
  label 块本质是无参函数，call/return 机制完全复用
```

### 5.3 goto（无条件跳转）

```
goto(label)

  resolveLabel → call(label)
  等价于 br 去掉条件判断，直接进入目标块帧
```

### 5.4 高级语法降级（lower 层职责，非 kvcpu 职责）

`if / for / while / break / continue` 由编译器在写入 `/func/` 前展开，
kvcpu 不感知这些语法，只执行降级后的四条原语（`call / return / br / goto`）：

| 高级语法 | 降级为 |
|---------|-------|
| `if(cond) { A } else { B }` | `br(cond, blockA, blockB)` |
| `for(cond) { body }` | 循环块 + 块尾 `br(cond, loopHead, loopEnd)` |
| `while(cond) { body }` | 同上 |
| `break` | `goto` 跳至循环块的 return 点 |
| `continue` | `goto` 跳至循环块头部 |

等价于 JVM 编译器将 `if`/`for`/`while` 编译为 `goto`/`ifeq`，字节码层不存在这些指令。

## 六、错误传播模型

kvcpu 采用**错误终止 + 调用栈展开（unwinding）**模型，类比 BEAM 的进程崩溃隔离：

```
错误触发（任意层）：
  vthread.SetError(vtid, pc, {code, message})
  status = "error"

调用栈展开：
  从当前 PC 开始，逐层 DelR 各帧，直到顶层
  （等价于 JVM uncaught exception：展开所有 Stack Frame）

错误记录：
  /vthread/<vtid>       = {"status":"error","pc":"...","error":{code,msg}}
  /vthread/<vtid>/done  = {"status":"error","error":{code,msg}}   （通知等待方）
```

**错误不跨 vthread 传播**。一个 vthread 的错误不影响其他 vthread——
等价于 BEAM 的进程隔离（Let it crash）。

调用方若需感知子 vthread 的错误，通过 `kv.Watch(/vthread/<vtid>/done)` 读取结果。

## 七、vthread 完整生命周期

```
① 创建
  kv.Set(/vthread/<vtid>, {pc:"/vthread/<vtid>/[0,0]", status:"init"})
  kv.Set(/vthread/<vtid>/[0,0], funcName)     // 入口函数
  kv.Notify(/vthread/ready, {vtid})           // 唤醒 worker

② 调度（worker pick）
  worker 原子将 status: init → running

③ 执行
  Fetch-Decode-Execute 循环直到 done / error / wait

④ 等待（异步 op）
  status = "wait"；op-plat 完成后写 /vthread/<vtid>/done
  worker 收到通知后将 status: wait → running，继续执行

⑤ 完成
  kv.Set(/vthread/<vtid>/done, {status:"ok", result:...})
  kv.Set(/vthread/<vtid>, {status:"done"})
  调用帧全部已被 DelR 清理

⑥ 回收（GC）
  done/error 状态的 vthread 由外部 GC goroutine 定期清理：
  kv.DelR(/vthread/<vtid>)
  时机：caller 已读取 done 结果，或超时（TTL）
```

## 八、并发内存模型

### 8.1 路径所有权

| 路径前缀 | 所有者 | 并发写 |
|---------|-------|-------|
| `/vthread/<vtid>/` | 单一 worker | **禁止**跨 vthread 写（热注入除外） |
| `/func/` | 编译器/热替换 | **允许**（不影响已在执行的帧） |
| `/sys/` | VM 管理 | 各 VM 写自己的 `/sys/vm/<id>/` |
| `/dev/` | 设备层 | 由设备驱动写入 |

### 8.2 原子性边界

kvspace 的单条 `kv.Set` 是原子操作（Redis 单命令原子）。
跨多条命令的操作不原子——kvcpu 依赖以下约定保证正确性：

- **Pick 竞争**：多 worker 同时 Set status=running，后写者赢，先写者已在执行不会重取
- **帧销毁**：`DelR` 是原子清理，不存在"帧半删"状态
- **vthread 状态变更**：只有持有该 vthread 的 worker 才写其路径（所有权隔离）

### 8.3 热代码替换

`/func/<pkg>/<name>` 被替换时：
- **已在执行的帧**：帧通过 Link 持有代码引用；Link-based 模型下，
  Link 解析在帧创建时完成，替换不影响正在运行的帧（当前 Copy 模型已在帧内有副本，同样不影响）
- **新 call**：将使用新版本代码（Link 到新路径）
- **一致性边界**：单个 vthread 在单次函数调用期间使用一致的代码版本

## 九、调度器

```
RunWorker(id):
  loop:
    pc = pick()             // 原子抢占 init vthread，返回绝对 PC
    if pc == "":
      wait()                // BLPOP /vthread/ready，避免空轮询
      continue
    Execute(ctx, pc)
```

**pick 原子性**：依赖 Redis 单线程——`Set(status=running)` 原子执行，
两 worker 竞争时后写者覆盖，但先写者已离开 init 状态，不会被重复拾取。
这是乐观并发，等价于 CAS（Compare-And-Swap）。

**wait 语义**：`kv.Watch(/vthread/ready, timeout)` 超时后重新 pick，
防止 Notify 在 pick 和 watch 之间到达导致通知丢失（time-of-check/time-of-use 保护）。

**公平性**：pick 扫描 /vthread/ 无优先级，先到先得。
若需优先级调度，可引入 `/vthread/ready-hi` / `/vthread/ready-lo` 两级队列（扩展点，不修改核心）。

## 十、栈保护

| 保护项 | 机制 |
|-------|------|
| 最大调用深度 | Execute 循环检测 PC 路径段数；超过 `MaxStackDepth`（默认 512）触发 stack overflow error |
| 无限循环检测 | 不做（kvlang 程序员负责；可通过外部 kvspace 监控 PC 是否长时间不变来检测） |
| 帧泄漏保护 | done/error 后 GC goroutine 强制 `DelR(/vthread/<vtid>)` 清理所有残留帧 |

## 十一、对外接口（CPU interface）

kvcpu 对外只暴露一个 interface。

```go
// CPU 是 kvcpu 对外的唯一接口。
type CPU interface {
    // Execute 同步执行一个 vthread 直到 done/error。
    // pc 为绝对路径：/vthread/<vtid>/[0,0]
    Execute(ctx context.Context, pc string) error

    // RunWorker 启动第 id 个 worker daemon，直到 ctx 取消。
    RunWorker(ctx context.Context, id int)
}

// New 创建标准 CPU 实现。
func New(kv kvspace.KVSpace) CPU
```

设计约束：
- `pick` / `wait` 是 `cpu` 结构体的私有方法，不出现在 interface
- interface 可 mock，利于单元测试和替换执行引擎
- 调用方从不感知调度细节

## 十二、文件职责

| 文件 | 职责 | 禁止 |
|------|------|------|
| `cpu.go` | `CPU` interface + `New` + `MaxStackDepth` 常量 | 实现逻辑 |
| `execute.go` | `Execute` 主循环 + dispatch switch | 硬编码路径字符串；import parser/lower/ast |
| `sched.go` | `pick`/`wait`（私有）+ `RunWorker` | 感知 opcode；执行指令 |
| `controlflow.go` | `handleControl`/`If`/`brToCall`/`resolveLabel` | import parser/lower；直接操作 kvspace 路径 |

```
cpu.go

```

kvcpu 是依赖链终点，不被任何内部包 import。

## 十三、禁止项

| 编号 | 禁止 | 理由 |
|------|------|------|
| B1 | dispatch 中做 KV 查询判断 opcode 类型 | 零查询是吞吐量底线；路由必须 O(1) 静态完成 |
| B2 | kvcpu import parser / lower / ast | 运行时不感知编译层；依赖方向单向 |
| B3 | 硬编码 kvspace 路径字符串 | 所有路径只由 keytree 生成 |
| B4 | vthread 状态变更绕过 `vthread.Set`/`vthread.SetError` | 必须有统一入口，确保状态持久化和可观测 |
| B5 | Pick 引入分布式锁 | Redis 单线程保证原子写；锁增加延迟和复杂度 |
| B6 | 执行器直接操作 kvspace（绕过 layoutcode/builtin/vtype） | 存储操作由各层封装；kvcpu 只做调度 |
| B7 | 将 `pick`/`wait` 导出为公开函数 | 调度原语是实现细节，不属于 CPU interface |
| B8 | 跨 vthread 写另一个 vthread 的私有路径 | 违反路径所有权原则；破坏并发安全 |
| B9 | 在 Execute 循环中不检查栈深度 | 无保护的递归会导致 KV 路径无限增长 |
