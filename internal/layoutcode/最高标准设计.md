# layoutcode 最高标准设计

---


## 设计目标

| 目标 | 含义 |
|------|------|
| **编译期写入，运行期只读代码** | WriteFunc 只写 /src/+/func/，HandleCall 只读 /func/ 签名 |
| **帧 = kvspace 子树** | call PC 即帧根，路径深度=调用栈深度，无需额外栈结构 |
| **零拷贝挂载（Link 模型）** | 目标：`kv.Link(/func/pkg/name, pc)` 替代 copyFunc 逐槽复制 |
| **块=无参函数** | br/goto 通过 FuncIdx 查找 label，复用 HandleCall 统一进入 |
| **原子帧销毁** | `kv.DelR(framePC)` 一次清理帧子树 |
| **两阶段分离** | WriteFunc 不碰 /vthread/，HandleCall/Return 不写 /func/ |

---

## 零、一句话定位

**layoutcode 是 kvlang 的 Linker**：编译期将 AST 链接为 `/func/` 中的平铺 KV 指令；
运行期将函数体挂载到 vthread 帧、拆卸帧。

类比：`ld`（静态，将 `.o` 写入可执行段）+ 动态加载器（运行时，将 `.so` 映射到进程地址空间）。

---

## 一、系统位置

```
parser/ast ──► lower ──► layoutcode.WriteFunc ──► /func/  （静态，一次写入，永久不变）

                              ▲  只读签名
kvcpu.handleControl ──► layoutcode.HandleCall   ──► /vthread/<vtid>/[pc]/   （动态，每次 call）
                    └─► layoutcode.HandleReturn ──► kv.DelR(frame)          （动态，每次 return）
```

layoutcode 是唯一**同时**操作 `/func/`（编译期写）和 `/vthread/`（运行期读写）的包。
两个阶段的代码在同一文件；边界由调用时机而非文件划分。

## 二、静态阶段：/func/ 写入

### 2.1 WriteFunc — 五步写入协议

```
WriteFunc(kv, pkg, *ast.Func):
  ① kv.Set( Src(pkg, name),     fn.FullText()    ) → /src/<pkg>/<name>           源码原文
  ② kv.Set( Func(pkg, name),    fn.Sig.String()  ) → /func/<pkg>/<name>          编译签名
  ③ WriteBody(...)                                  → /func/<pkg>/<name>/[i,j]   指令体
  ④ RegisterBlocks(...)                             → /func/<pkg>/<name>/<label>/ 块标签
  ⑤ kv.Set( FuncIdx(name),      pkg              ) → /func/idx/<name>            反向索引
```

### 2.2 指令体写入（对齐 keytree §三）

`WriteBody` 将 `[]ast.Stmt` 线性展开，`*ast.Instruction` 写三组 KV：

```
/func/<pkg>/<name>/[i, 0]    opcode          必须
/func/<pkg>/<name>/[i,-1]    reads[0]        reads 从 -1 递减
/func/<pkg>/<name>/[i,-2]    reads[1]
/func/<pkg>/<name>/[i,+1]    writes[0]       writes 从 +1 递增
```

函数体共用单一 `idx` 计数器（单调递增）；`*ast.BlockStmt` 子树独立计数（各自从 0 起）。

### 2.3 块标签注册（对齐 kvcpu §5.2）

`lower` 将 `if`/`for` 降级为 `*ast.BlockStmt`，`RegisterBlocks` 将其注册为无参子函数：

```
/func/<pkg>/<parent>/<label>    = "def <label>() -> ()"
/func/idx/<parent>/<label>      = pkg
```

`br`/`goto` 通过 `FuncIdx` 查找，复用 `HandleCall` 统一进入——块跳转与函数调用同一条路径。

## 三、运行时阶段：帧管理

### 3.1 帧路径约定

`call` 指令的绝对 PC **就是**帧根目录（无需额外路径拼接）：

```
call PC = /vthread/42/[3,0]
帧根   = /vthread/42/[3,0]          ← 即 call PC 本身
跳入点  = /vthread/42/[3,0]/[0,0]   ← 返回值 pc + "/[0,0]"
帧变量  = /vthread/42/[3,0]/x       ← ./x 解析为 帧根/x
```

路径深度 = 调用栈深度；无需额外栈结构（对齐 kvcpu §二 §2.2）。

### 3.2 HandleCall 语义

**当前实现（Copy 模型）**

```
HandleCall(ctx, kv, pc, inst, tail):
  pkg = FuncIdx(inst.Reads[0])              1 次 KV 查询
  sig = ParseFuncSig(kv.Get(funcKey))       1 次 KV 查询 + parse
  bindings = { 形参名 → 实参路径, ... }
  copyFunc(funcKey, pc, bindings)           逐槽复制 + 值替换（临时实现，见 §三.3）
  追加 return 兜底
  return pc + "/[0,0]"
```

**标准模型（Link-based，P4，对齐 kvcpu §5.1 和 keytree §8.3）**

```
HandleCall(ctx, kv, pc, inst, tail):
  pkg = FuncIdx(inst.Reads[0])
  kv.Link(/func/<pkg>/<name>, pc)           零拷贝挂载代码
  for i, param := range sig.ParamNames():
      kv.Set(pc+"/"+param, inst.Reads[i+1]) 绑定实参
  return pc + "/[0,0]"
```

P4 完成后 `copyFunc` 整体删除，HandleCall 缩减为 ~20 行。

**参数槽位协议：**

```
inst.Reads[0]      funcName（非实参）
inst.Reads[1..N]   第 1..N 个实参 → funcSig.ParamNames()[0..N-1]
inst.Writes[0..M]  第 1..M 个返回槽 → funcSig.ReturnNames()[0..M-1]
```

`tail bool` 预留尾调用优化（TCO）入口——TCO 复用父帧而非创建新子树，当前未启用。

### 3.3 HandleReturn 语义

```
HandleReturn(ctx, kv, pc, inst) → (nextPC, retVal):

  framePC = pc 去掉末尾 [i,0] 段

  ── 顶层 return（帧根 == vthread 根）──────────────────────────────
  retVal = kv.Get(framePC + "/" + inst.Reads[0][2:])   读返回变量
  return ("", retVal)
  ← 调用方（kvcpu）凭此调 vthread.SetDone(vtid, retVal)

  ── 嵌套 return ────────────────────────────────────────────────────
  v = kv.Get(framePC + "/" + srcVar)
  kv.Set(parentFrame + "/" + dstVar, v)    写回父帧目标槽
  kv.DelR(framePC)                         一次销毁帧（Link + 所有局部槽）
  return (NextPC(framePC), "")
  ← 调用方调 vthread.Set(vtid, nextPC, "running")
```

**PC 契约**：HandleReturn 不直接写 `.pc`；只返回 `nextPC` 或 `retVal`，由 `kvcpu/controlflow.go` 持有后写入（对齐 vthread 包 PC 更新契约）。

## 四、两阶段边界

| 函数 | 操作路径 | 调用时机 | 是否可并发 |
|------|---------|---------|-----------|
| `WriteFunc` | `/src/` 写、`/func/` 写 | 编译期（`kvlang load`） | 允许：代码是 append-only |
| `HandleCall` | `/func/` 只读签名；`/vthread/` 写帧 | 运行时，持有 vthread 所有权 | 一个 vtid 只有一个 worker |
| `HandleReturn` | `/vthread/` 读返回值；`DelR` 帧 | 运行时，持有 vthread 所有权 | 同上 |

`WriteFunc` 从不接触 `/vthread/`；`HandleCall`/`HandleReturn` 从不写 `/func/`。

---

## 五、文件职责

单文件 `layoutcode.go`，当前 ~250 行；P4 完成删除 `copyFunc` 后 ≤ 150 行。

| 函数 | 阶段 | P4 后状态 |
|------|------|---------|
| `WriteFunc` | 静态 | 不变 |
| `WriteBody` / `writeStmt` | 静态 | 不变 |
| `RegisterBlocks` | 静态 | 不变 |
| `HandleCall` | 运行时 | 改为 Link；代码量从 ~55 行降至 ~20 行 |
| `HandleReturn` | 运行时 | 不变（`DelR` 已是原子操作） |
| `copyFunc` | 临时 | **整体删除** |

## 六、禁止项

| 编号 | 禁止 | 理由 |
|------|------|------|
| LC1 | `WriteFunc`/`WriteBody` 读写 `/vthread/` | 静态写入不感知运行时 |
| LC2 | `HandleCall`/`HandleReturn` 写 `/func/` | 运行时不修改代码段 |
| LC3 | layoutcode 直接写 `.pc`（keytree.VThreadPC） | PC 更新契约归 kvcpu；layoutcode 只返回 nextPC 由调用方写入 |
| LC4 | 调用 `vthread.SetDone`（正常完成路径） | 顶层 return 的完成语义由 kvcpu 决策，layoutcode 只上报 retVal |
| LC5 | layoutcode 硬编码路径字符串 | 所有路径由 keytree 函数生成 |
| LC6 | 扩展 `copyFunc` 逻辑 | copyFunc 是待删除的临时实现，不应承载新逻辑 |
